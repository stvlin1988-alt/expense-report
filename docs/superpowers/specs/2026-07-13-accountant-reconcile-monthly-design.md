# 會計核銷 ＋ 月結 設計（Phase 1）

日期：2026-07-13
來源：`docs/accountant-meeting-2026-07-reconcile-and-monthly.md`（議題單）與會計討論結論。
狀態：設計定案，待寫實作計畫。

系統現況：門市 PWA 拍單 → Gemini OCR → 員工確認送出（`submitted`）→ 店主管稽核打勾（`audited`）。
本設計接在 `audited` 之後：**會計核銷 → 月結封月**。

---

## 1. 決策摘要（會計討論結論）

| 項目 | 決策 |
|---|---|
| 核銷本質 | 逐筆通過／退回；另提供勾選後一鍵核銷 |
| 核銷顆粒度 | 逐筆；畫面要有日小計與期間合計 |
| 退回 | 退給該店主管，原因必填；**員工端完全不露出** |
| 退回迴路 | 主管改完重送 → 回會計待核銷 |
| 燈號 | 會計端沿用主管的 `audit_light`，唯讀；會計任何動作都不改燈 |
| 核銷後修改 | 主管不能直接改已核銷單；要改由**會計退回** → 主管改 → 重送 → 再核銷 |
| 會計新增單 | 可新增（摘要／金額／科目／選門店／選日期），建出來直接是已核銷 |
| 會計科目 | 沿用現有兩層分類（附錄 A），**不做額外對應表**；發票／收據／無單據不分流 |
| 待核銷範圍 | 只看 `audited` 的單，主管沒打勾的會計看不到 |
| 會計可見範圍 | 跨店 |
| 月結週期 | 由「月結日」定義，**半開區間**：月結日 D 當天開新期，結的是 D 之前（不含 D） |
| 月結日調整 | 可調（農曆年提早／延後），只有經理能設 |
| 月報表 | 分店 × 科目大類的交叉表，大類可展開細類；不做匯出、不做上期比較、不做點到單據明細 |
| 未核銷單（已打勾） | 期間進行中時計入報表但分開標示；封月時自動挪到下一期繼續核 |
| 主管沒打勾的單 | 封月時**不挪期、不進任何帳**，留在已封期永久唯讀；會計端另列「上期未處理單」供複查，要認就在下一期手動新增一筆 |
| 封月 | 期間結束後有寬限期，期間內員工／主管／會計三方都還能對上一期作業；寬限期滿自動封，會計可提前手動封 |
| 反封月 | **沒有**。封月後該期永久唯讀 |
| 沖帳／負數金額 | Phase 1 不做，金額維持正數 |
| 經理 | **只看月結報表 ＋ 設定月結日**，完全不碰核銷 |

## 2. 資料模型

### 2.1 新表 `accounting_periods`（會計期間）

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | int PK | |
| label | str | 期間名稱，例如 `2026-01`。依**起始日所屬月份**命名 |
| start_date | date | 起始營業日（含） |
| end_date | date | 結束營業日（含） |
| lock_at | datetime (UTC) | 鎖定時刻。到此時間自動封月 |
| status | str | `open` / `closing` / `closed` |
| closed_by | int FK users | 手動封月者；自動封月為 null |
| closed_at | datetime (UTC) | |

- 期間**首尾相接、不重疊、不留洞**；`start_date` = 前一期 `end_date` + 1 天。
- `status` 語意：
  - `open`：今天落在 `[start_date, end_date]` 之內。
  - `closing`：期間已結束、還沒到 `lock_at`。**寬限期**——會計仍可對本期動作。
  - `closed`：已封月。該期所有單據永久唯讀。
- 期間由設定推導、自動延展；不需要人工每月開期。

### 2.2 新設定（`app_settings`，key/value）

| key | 預設 | 說明 |
|---|---|---|
| `period_close_day` | `1` | 月結日。D 號當天開新期，結的是 D 之前（不含 D） |
| `period_lock_offset_hours` | `36` | 從換期日 00:00（台灣時間）起算多久後鎖定。36 小時＝隔天中午 12:00 |

- 只有經理能改。
- 農曆年調整：經理直接改**該期的 `end_date`**，下一期的 `start_date` 與 `lock_at` 自動跟著重算。

### 2.3 `expenses` 新增欄位

| 欄位 | 型別 | 說明 |
|---|---|---|
| `status` | 既有 | 新增兩個值：`reconciled`、`rejected` |
| `reconciled_by` | int FK users | |
| `reconciled_at` | datetime (UTC) | |
| `reject_reason` | str(200) | 退回原因，退回時必填 |
| `period_id` | int FK accounting_periods | **認列期間**。預設依 `business_date` 落到所屬期；會計按「挪下期」即指向下一期 |

「挪下期」＝ 改 `period_id`，**不作廢也不重開新單**。原圖、OCR 原始結果、log 全部保留。

### 2.4 軌跡

沿用現有 `audit_log`，新增 action：`reconcile`、`unreconcile_reject`、`edit_amount`、`edit_category`、`move_period`、`period_close`。不另開表。

## 3. 狀態機

```
submitted ──主管打勾──> audited ──會計核銷──> reconciled
                          ▲                       │
                          │                       │ 會計退回（要改帳時）
                     主管改完重送                  │
                          │                       ▼
                          └──────────────────  rejected
                                    ▲
                                    └── 會計退回（待核銷時）
```

- 允許的邊，只有這些：
  - `audited → reconciled`（核銷）
  - `audited → rejected`（退回，原因必填）
  - `reconciled → rejected`（已核銷但要改帳 → 退回給主管）
  - `rejected → audited`（主管改完重送）
- 會計新增的單：直接建成 `reconciled`（不回頭走主管打勾）。`has_receipt=False`，燈號因此是黃的（現有 `audit_light` 對無單據即黃），一眼看得出是會計補的帳。
- `submitted` 不能直接跳 `reconciled`（主管沒打勾會計看不到）。
- **任何寫入，若目標單的 `period.status == closed`，一律擋掉。**

## 4. 會計端 UI（核銷）

新分頁「核銷」，只有 **會計** 看得到（經理不碰核銷）。

**篩選**：期間（預設當期）、門店（全部／單店）、狀態（待核銷／已核銷／退回中／全部）、科目、日期區間。

**合計列**：隨篩選即時變動，顯示筆數、總額，並拆成「已核銷 $X ／ 待核銷 $Y」。

**清單**：依營業日分組，每組一列**日小計**。每列欄位——營業日、單號、門店、燈號（唯讀）、摘要、科目、金額、單據縮圖（無單據標示）、狀態。**金額與科目就地可改**，改動自動留 log。

**動作**：
- 逐筆：核銷 ／ 退回（跳原因輸入，必填）／ 挪下期
- 批次：勾選多筆 → 一鍵核銷（僅核銷，退回與挪期維持逐筆）
- 新增單據：選門店＋日期，填摘要／金額／科目 → 直接建成已核銷

**「上期未處理單」清單**（唯讀）：已封期裡主管沒打勾的單，列出門店、營業日、金額、摘要、原圖。會計據此判斷要不要認帳；要認就用上面的「新增單據」在本期補一筆。這份清單只有會計看得到。

單據原圖走現有加密 R2 簽名網址；會計跨店，看得到所有門店的圖。

## 5. 月結

### 5.1 期間與封月時間軸（以月結日 = 1 號為例）

```
1/1 ────────── 1/31        2/1              2/2 12:00
│   1月期間（open）  │  寬限期（closing）  │   已封月（closed，永久）
                     └── 新的 2 月期間同時開始
```

- 2/1 一到：1 月期間結束、2 月期間開始。1 月期間進入 `closing`。
- **寬限期（2/1 ~ 2/2 中午）三方都還能對 1 月作業**——這是必要的，因為營業日 08:00 分界代表 **1/31 的營業日要到 2/1 早上 08:00 才結束**，門市在 2/1 早上本來就還在處理 1/31 那批單：
  - **員工**：送出營業日屬於 1 月的單（含 2/1 凌晨拍、營業日算 1/31 的）。
  - **主管**：稽核打勾、交接班、改被退回的單並重送。
  - **會計**：核銷／退回／挪期／新增單據。
- 單據落在哪一期看**營業日**，不看拍照時間：2/1 早上 08:00 之後拍的單營業日是 2/1，自動落到 2 月期間，不會汙染 1 月。
- 2/2 12:00：自動封月，三方同時鎖。
- 會計可在寬限期內**提前手動封月**（二次確認，不可逆）——提前封就是提前把門市也鎖掉，所以按之前要看清楚該期還有沒有沒打勾的單。

### 5.2 自動封月的實作

**不用背景排程**（符合專案「前端不輪詢、狀態全進 DB」的原則）：任何請求碰到某一期時，同步檢查 `now >= lock_at`，若成立就在**同一個交易**裡：

1. 把該期**已打勾但沒核銷完**的單（`audited` / `rejected`）`period_id` 改到下一期（留 log）。
2. 該期**沒打勾**的單（`submitted`）**留在原期不動**（見 §5.4）。
3. 期間 `status` 設為 `closed`。

以資料庫層條件更新（`WHERE status != 'closed'`）確保多 worker 同時碰只會封一次。

### 5.4 主管沒打勾的單（`submitted`）

主管每天都會稽核，「拖著沒打勾」是門市管理問題。封月時**兩種殘留單走不同的路**：

| 殘留單 | 封月時 | 理由 |
|---|---|---|
| `audited` / `rejected`（已打勾、會計還沒核完） | 挪到下一期，會計下期繼續核 | 這筆錢遲早要進帳，只是進下一期 |
| `submitted`（主管根本沒打勾） | **不挪期、不進任何帳** | 封月了就不動帳；沒經過門市覆核的錢不自動認列 |

沒打勾的單封月後的處置：

- 留在已封期，**永久唯讀**——主管不能再補打勾，也不會進任何一期的帳。
- 會計端另有一份**「上期未處理單」清單**（門店、營業日、金額、摘要、原圖），讓會計自己判斷要不要認這筆錢。
- 會計要認的話，就在**下一期用「新增單據」手動補一筆**、自己指定科目——那筆帳掛在下一期、由會計自己開，軌跡清楚。
- **主管端逾期提醒**（封月前的預防）：營業日已過一天、仍是 `submitted` 的單，在主管稽核區顯著標示（「有 N 筆昨天以前的單還沒打勾」），逼門市在封月前處理掉。主管在寬限期內（換期日到鎖定時刻）還有一天半可以補打勾，所以真的殘留到封月的應該是極少數。

### 5.3 月報表

**只有會計與經理看得到，主管與員工完全看不到。**

選定一期後呈現交叉表：

- **列**：科目大類（可展開為細類）
- **欄**：門店 ＋ 最右「總計」欄
- **底列**：各店合計
- 期間為 `open` / `closing` 時，每格拆成「已核銷 ／ 待核銷」兩個數字；`closed` 之後待核銷必為 0，只顯示單一數字。

不做：Excel／CSV 匯出、與上期比較、點格子下鑽到單據明細。（會計明確表示畫面看就好。）

## 6. 權限

| 角色 | 核銷清單 | 核銷／退回／挪期／新增單 | 月報表 | 封月 | 月結日設定 |
|---|---|---|---|---|---|
| 員工 | ✗ | ✗ | ✗ | ✗ | ✗ |
| 主管 | ✗ | ✗（只在稽核區改被退回的單） | ✗ | ✗ | ✗ |
| 會計 | ✓ 跨店 | ✓ | ✓ | ✓ | ✗ |
| 經理 | ✗ | ✗ | ✓ | ✗ | ✓ |

- `accountant` 角色**已存在**於 `User.ROLES`，後台已能建會計帳號，不需新增角色。
- 會計碰不到後台的帳號／門店／裝置管理。
- 認證沿用現行機制：綁定裝置（綁定碼）＋ 密碼 ＋ 人臉。會計的辦公室電腦走同一套綁定流程。
- 員工端與主管端的既有畫面**不改**；退回狀態不對員工露出（延續現行「複查不洩漏主管改動」的白名單設計）。

## 7. 邊界與錯誤處理

- **已封月**：任何寫入（核銷、改金額、改科目、退回、挪期、會計新增單日期落在該期）一律拒絕，回明確錯誤而非預設放行。**主管對已封期的 `submitted` 單打勾也要擋掉**——封月後不動帳。
- **挪下期**：下一期不存在時自動延展生成；下一期已封月則拒絕。
- **併發核銷**：兩個會計同時核同一筆，後到者必須看到「已被核銷」而非覆蓋——用資料庫層條件更新（`WHERE status = 'audited'`）擋。
- **併發封月**：見 §5.2，條件更新確保只封一次。
- **期間完整性**：改 `end_date` 時檢查不得與相鄰期重疊或留洞，且不得改已封月的期間。
- **退回原因**：必填，長度上限 200；空白或純空格拒絕。
- **金額**：維持正數（Phase 1 不做負數沖帳單）。

## 8. 測試

- **狀態機**：每一條合法邊各一個測試；**不合法的邊要失敗**（`submitted → reconciled`、`draft → reconciled`、主管改已核銷單）。
- **封月**：寬限期內**員工能送出、主管能打勾與交班、會計能核銷**（三方都要各一個測試），鎖定後三方一律拒寫；封月只把 `audited` / `rejected` 挪到下一期，`submitted` 留在原期且封月後打勾要被拒；併發只封一次。
- **營業日與期間歸屬**：2/1 早上 08:00 前拍的單營業日為 1/31、落在 1 月期間；08:00 後拍的落在 2 月期間。
- **上期未處理單**：已封期的 `submitted` 單出現在會計的清單裡，且不計入任何一期的報表金額。
- **逾期提醒**：營業日已過一天仍為 `submitted` 才提醒；當天的單不提醒。
- **期間切割**：預設月結日的半開區間正確（1 號那天算新期）；農曆年調 `end_date` 後相鄰期首尾相接；期間跨月時營業日 08:00 分界仍正確。
- **權限範圍**：會計看得到全部門店、主管只看得到自己店、經理看得到報表但碰不到核銷、員工看不到退回狀態。
- **報表加總**：分店 × 科目大類的合計與明細相符；已核銷／待核銷拆分正確。
- 後端 `python3 -m pytest -q`；前端純邏輯 `node --test tests/js/*.mjs`。
